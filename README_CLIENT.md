# Client Guide: Local LLM Test Case Generator

Краткая инструкция для клиента: как открыть демо, проверить генерацию тест-кейсов и интерпретировать результат.

## 1. Вход в сервис
Откройте ссылку, которую вам передал владелец сервиса:
- `https://<demo-host>/docs`

Это страница Swagger UI со всеми доступными методами.

## 2. Проверка за 5 минут

### Шаг 1. Проверить доступность сервиса
Метод: `GET /health`

Ожидаемый результат:
- код `200`
- `status: "ok"`

### Шаг 2. Загрузить документ
Метод: `POST /upload`

Действия:
- нажать `Try it out`
- выбрать PDF с требованиями/ТЗ
- нажать `Execute`

Ожидаемый результат:
- код `200`
- ответ содержит `file_id` (его нужно использовать в шаге 3)

### Шаг 3. Сгенерировать тест-кейсы
Метод: `POST /generate`

Пример тела запроса:
```json
{
  "file_id": "<FILE_ID_FROM_UPLOAD>",
  "user_prompt": "Сформируй 12 атомарных тест-кейсов: 4 positive, 4 negative, 4 boundary, без дублей, с конкретным expected_result",
  "top_k": 12,
  "max_cases": 12,
  "include_negative": true,
  "include_boundary": true
}
```

Ожидаемый результат:
- код `200`
- в ответе есть `test_cases`
- в ответе есть `quality_report`

## 3. Оценка качества генерации (опционально)
Метод: `POST /evaluate`

Передайте:
- `generated_cases`: массив `test_cases` из ответа `/generate`
- `reference_cases`: эталонные кейсы (если есть), иначе `[]`

Минимальный пример:
```json
{
  "generated_cases": [
    {
      "title": "Пример кейса",
      "preconditions": "Данные загружены",
      "steps": ["Открыть форму", "Нажать отправить"],
      "expected_result": "Запрос успешно обработан",
      "scenario_type": "positive",
      "source_ref": "page_1_offset_100",
      "priority": "medium"
    }
  ],
  "reference_cases": []
}
```

Ожидаемый результат:
- код `200`
- блок `metrics` с итоговыми метриками качества

## 4. Как трактовать результат
Хороший результат:
- есть баланс `positive`, `negative`, `boundary`
- у каждого кейса есть понятные шаги и проверяемый `expected_result`
- кейсы не дублируются
- `traceability_rate` близок к `1.0`

## 5. Частые ошибки и решение
- `422 json_invalid`
  Причина: в теле запроса невалидный JSON (часто переносы строк внутри строки).
  Решение: вставить валидный JSON, проверить кавычки и запятые.
- `404 file_id not found`
  Причина: устаревший или неверный `file_id`.
  Решение: заново выполнить `POST /upload` и взять новый `file_id`.
- `422 Model returned incomplete testcase fields`
  Причина: модель вернула неполную структуру кейсов.
  Решение: повторить `/generate` с более строгим `user_prompt` и меньшим `max_cases` (например, 8-10).
- `502`
  Причина: локальная LLM недоступна.
  Решение: обратиться к владельцу сервиса.

## 6. Важные ограничения
- Решение работает в демо-режиме и не заменяет финальное QA-ревью.
- Не загружайте чувствительные/персональные данные без согласования.
